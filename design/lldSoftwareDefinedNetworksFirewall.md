## Introduction

## Purpose

The purpose of this document is to provide detailed design required to implement validated model of a NSX-T Distributed Firewall in accordance with Atos standards and portfolio services. The principal aim of this document is to translate the high-level design (HLD) into a technical low-level design (LLD).

Document is divided into two logical pieces. First related with Management Domain Distributed Firewall configuration and second related with Workload Domain Distributed Firewall configuration. Each of these sections is covering configuration for Services, Security Groups and DFW ruleset.

## Audience

This document is intended for Atos Cloud Services Engineers and Architects responsible for VMware Cloud Services (VCS) solution implementation and maintenance.

## Scope

That LLD is intended to cover below components and domains:

1. VCS DFW ruleset
2. VCS Security Groups
3. VCS Security Services

The following areas are explicitly out of scope:

1. Architecture overview
2. Other NSX-T configuration apart of DFW
3. Scalability Design
4. Availability Design
5. Customer related ruleset
6. Underlay Network configuration
7. External Network configuration
8. IP address management

## Distributed Firewall Ruleset

### Management Domain DFW ruleset

This section is covering list of the rules implemented under Management Domain DFW. Rules are reflecting current configuration allowing only necessary traffic within Management Domain. Each rule is represented in below table by multiple lines where differentiator is **RuleName**. When rule name is duplicated in next few lines, it means that rule is containing all details from those lines. Apart of **RuleName**, rules are having information about **SectionName** used to group rules together into logical blocks, **Source** and **Destination** which are showing what is the source and destination of the traffic, **Service** which is used to define services required for this particular traffic, **Action** is determining if traffic is Allowed, Dropped or Rejected and **ApplyTo** which is defining scope of the rule (where it is implemented).

Following table is using two types of Security Groups. Standard one, which is based on IP addressed and Dynamic one which is based on dynamic criteria to match VMs based on the name. Differentiation between these two is inside name of the security Group, where Dynamic one is having **_APPLYTO** postfix in the name.
