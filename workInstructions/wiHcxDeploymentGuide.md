# Network Extension

HCX Network Extension (HCX-NE), you can extend the virtual machine (VM) networks to an HCX remote site. Virtual machines that are migrated or created on the extended segment at the remote site behave as if on the same L2 segment as virtual machines in the source environment. With Network Extension, the default gateway for the extended network is only connected at the source site. Traffic from virtual machines in remote sites that must be routed to a different L3 network flow through the source site gateway.

Using Network Extension with VMware HCX Migration you can:

- Retain the IP and the MAC addresses of the VM, and honor existing network policies.
- Extend VLAN-tagged networks from a VMware vSphere Distributed Switch.
- Extend NSX segments.

## Requirements

- Never extend the networks used to create the network profiles.
- Never use HCX to extend the vSphere Management network or other VMkernel networks (for example: vMotion, vSAN, replication) to the remote site.

# HCX Migration

Multiple migration techniques can be used to transfer virtual machines across data centers with HCX activation. The various migration steps are described in the following steps.

# Reqirements

- HCX Interconnect tunnels must be up/active.
- HCX vMotion requires 100 Mbps or higher throughput capability.
- Hybrid Interconnect, Bulk Migration, vMotion, and Replication Assisted vMotion services must be activated and in a healthy state in the relevant Service Mesh.

# Vmotion-Migration

VMware HCX vMotion can transfer a live Virtual Machine from a  vCenter Server with HCX activation to a destination site with HCX activation (or viceversa). The virtual machine's active memory, execution status, IP address, and MAC address are all recorded throughout the vMotion transfer. The connectivity, including the available bandwidth and the latency between the two locations, will determine how long the transfer will take.

# Replication-assisted VMotion

HCX Replication Assisted vMotion (RAV) uses the HCX along with replication and vMotion technologies to provide large scale, parallel migrations with zero downtime.

Replication Assisted vMotion creates two folders at the destination site. Both the virtual machine disk information and the definition of the virtual machine architecture are included in separate folders. The functioning of the virtual machine at the destination location is unaffected by this typical RAV migrating behavior.

# Bulk-Migration

Bulk migration uses the host-based replication to relocate a virtual machine between HCX data centers.

To reduce the downtime, the source VM remains online during the replication and is bootstrapped on the destination ESX host after replication completes.

## Analysis of CI configuration modifications following VM migration to VMC

- The virtual machine can be viewed in **Service Now** under the VMC VCenter after being migrated from on-prem to VMC.
- Since discovery has been enabled, if we modify CPU, Disk, or Memory during migration, the modified values will be reflected in Service Now as well.
- The VCenter Reference and Vcenter UUID will be changed since the VM has been moved to the VMC.
- After migration, the VM UUID and BIOS UUID will both stay the same.
